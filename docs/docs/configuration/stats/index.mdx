# Stats API

Cleanuparr exposes a stats endpoint with aggregated application statistics, designed for integration with dashboard tools like [Homepage](https://gethomepage.dev), [Homarr](https://homarr.dev), or any tool that can consume JSON APIs.

<Warning>
**`/api/stats` (v1) is deprecated.** Use **`/api/v2/stats`** instead. The v1 endpoint still works but returns `Deprecation`, `Sunset`, and `Link` response headers and is scheduled for removal after **2026-08-01**. See the **Deprecated: v1** section at the bottom of this page for the migration.
</Warning>

<ConfigSection
  title="API Key"
>

This endpoint requires authentication. You can authenticate using your API key in one of two ways:

**Header (recommended):**
```bash
curl -H "X-Api-Key: YOUR_API_KEY" http://localhost:11011/api/v2/stats
```

**Query parameter:**
```bash
curl http://localhost:11011/api/v2/stats?apikey=YOUR_API_KEY
```

You can find your API key in the Cleanuparr UI under **Account Settings**.

<Note>
JWT Bearer tokens are also supported for browser-based access.
</Note>

<SectionTitle>Endpoint</SectionTitle>

<ConfigSection
  title="GET /api/v2/stats"
>

Returns aggregated statistics for a time window. Every section except `health` is scoped to the window; `health` is a cached, point-in-time gauge. By default the windowed sections count only **live** activity — dry-run activity is excluded unless you opt in.

**Query Parameters:**

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `hours` | integer | `168` | Timeframe in hours (range: 1–8760) |
| `includeDryRun` | boolean | `false` | When `true`, dry-run activity is included in the windowed sections |

**Examples:**

```bash
# Last 7 days (default)
curl -H "X-Api-Key: YOUR_API_KEY" http://localhost:11011/api/v2/stats

# Last 24 hours
curl -H "X-Api-Key: YOUR_API_KEY" http://localhost:11011/api/v2/stats?hours=24

# Last 30 days
curl -H "X-Api-Key: YOUR_API_KEY" http://localhost:11011/api/v2/stats?hours=720

# Include dry-run activity
curl -H "X-Api-Key: YOUR_API_KEY" "http://localhost:11011/api/v2/stats?includeDryRun=true"
```

<Note>
If `BASE_PATH` is set, include it in the URL: `http://localhost:11011/cleanuparr/api/v2/stats`
</Note>

</ConfigSection>

<ConfigSection
  title="events"
>

The raw event audit for the window. The higher-level sections (`strikes`, `removals`, `cleaned`, `searches`) are ergonomic roll-ups derived from these same events.

- **totalCount** — Total events in the window (equal to the sum of `byType`)
- **byType** — Events grouped by event type
- **bySeverity** — Events grouped by severity (`Information`, `Warning`, `Important`, `Error`)

</ConfigSection>

<ConfigSection
  title="strikes"
>

Strike activity in the window.

- **issued** — Total strikes issued (equal to the sum of `byType`)
- **byType** — Strikes issued grouped by strike type (`Stalled`, `FailedImport`, `SlowSpeed`, `SlowTime`, `DownloadingMetadata`, `DeadTorrent`)
- **recovered** — Downloads that recovered and had their strikes reset

</ConfigSection>

<ConfigSection
  title="removals"
>

Downloads removed from the queue in the window, broken down by reason. This is the single source of truth for removals.

- **total** — Total removals in the window (all reasons)
- **byReason** — Removals grouped by delete reason (`Stalled`, `FailedImport`, `SlowSpeed`, `SlowTime`, `DownloadingMetadata`, `AllFilesSkipped`, `AllFilesSkippedByQBit`, `AllFilesBlocked`, `AtLeastOneFileBlocked`)

<Note>
**Malware blocked** is not a separate field — it is derived from this breakdown: `byReason.AllFilesBlocked + byReason.AtLeastOneFileBlocked`.
</Note>

</ConfigSection>

<ConfigSection
  title="cleaned"
>

Downloads cleaned by the download cleaner in the window. Cleaning is distinct from a removal — it happens when a download meets its seeding goals, not because it was struck out.

- **total** — Total cleaned downloads in the window
- **byReason** — Cleaned downloads grouped by clean reason (`MaxRatioReached`, `MaxSeedTimeReached`)

</ConfigSection>

<ConfigSection
  title="searches"
>

Seeker search activity in the window. A single record is tracked per search and updated as it completes, so the counts never double-count.

- **triggered** — Total searches triggered in the window (regardless of current status)
- **completed** — Searches that completed successfully
- **failed** — Searches that did not succeed (failed **and** timed-out searches)
- **grabbed** — Total items grabbed as a result of the triggered searches
- **byReason** — Searches grouped by the reason they were triggered (`Missing`, `QualityCutoffNotMet`, `CustomFormatScoreBelowCutoff`, `Replacement`)

</ConfigSection>

<ConfigSection
  title="jobs"
>

Scheduled job run outcomes in the window. Job runs are recorded regardless of dry-run mode.

- **totalRuns** / **completed** / **failed** — Overall run counts across all job types
- **byType** — Per-job-type stats (`QueueCleaner`, `MalwareBlocker`, `DownloadCleaner`, `BlacklistSynchronizer`, `Seeker`, `CustomFormatScoreSyncer`)
  - **totalRuns** / **completed** / **failed** — Run counts for this job type
  - **lastRunAt** — Timestamp of the most recent run
  - **nextRunAt** — When this job is next scheduled to run

</ConfigSection>

<ConfigSection
  title="health"
>

Current health of all configured download clients and arr instances.

- **downloadClients** — Health of each enabled download client
- **arrInstances** — Health of each enabled arr instance (Sonarr, Radarr, Lidarr, Readarr, Whisparr)

<Important>
`health` is a **gauge**, not a windowed metric. It reflects the last background check (~5 minute interval) and ignores the `hours` parameter, keeping the endpoint fast for frequent polling.
</Important>

</ConfigSection>

<ConfigSection
  title="windowHours & generatedAt"
>

- **windowHours** — The window the response covers, echoing the requested `hours` after clamping
- **generatedAt** — When the response was generated (UTC)

</ConfigSection>

<SectionTitle>Timeline</SectionTitle>

<ConfigSection
  title="GET /api/v2/stats/timeline"
>

Returns a bucketed time series for a single metric — useful for charts.

**Query Parameters:**

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `metric` | string | `events` | One of `strikesIssued`, `recovered`, `removed`, `malwareBlocked`, `events` |
| `hours` | integer | `720` | Timeframe in hours (range: 1–8760) |
| `bucket` | string | *(auto)* | Bucket size: `hour`, `day`, `week`, `month`. When omitted, defaults to `hour` for windows up to 24h and `day` otherwise |
| `includeDryRun` | boolean | `false` | When `true`, dry-run activity is included |

**Example:**

```bash
curl -H "X-Api-Key: YOUR_API_KEY" \
  "http://localhost:11011/api/v2/stats/timeline?metric=removed&hours=8760&bucket=month"
```

**Response:**

```json
[
  { "date": "2025-01-01T00:00:00Z", "count": 12 },
  { "date": "2025-02-01T00:00:00Z", "count": 8 },
  { "date": "2025-03-01T00:00:00Z", "count": 15 }
]
```

- **date** — Start of the bucket (UTC); granularity matches the requested `bucket`
- **count** — Count of the metric within that bucket

An unsupported `bucket` value returns `400 Bad Request`.

</ConfigSection>

<SectionTitle>Dashboard Integration</SectionTitle>

<ConfigSection
  title="Homepage Example"
>

[Homepage](https://gethomepage.dev) can use the `customapi` widget to display Cleanuparr stats:

```yaml
- Cleanuparr:
    icon: cleanuparr.png
    href: http://localhost:11011
    description: Arr Cleanup Service
    widget:
      type: customapi
      url: http://localhost:11011/api/v2/stats
      headers:
        X-Api-Key: YOUR_API_KEY
      mappings:
        - field: events.totalCount
          label: Events (7d)
          format: number
        - field: strikes.issued
          label: Strikes issued
          format: number
        - field: removals.total
          label: Removed
          format: number
        - field: cleaned.total
          label: Cleaned
          format: number
```

You can reference any field in the response using dot notation (e.g., `jobs.byType.QueueCleaner.failed`, `removals.byReason.AllFilesBlocked`).

To change the timeframe, add a query parameter to the URL: `http://localhost:11011/api/v2/stats?hours=720` for a 30-day view.

</ConfigSection>

<ConfigSection
  title="Other Dashboard Tools"
>

Any tool that can fetch and parse JSON can use this endpoint. The response is a flat, predictable JSON structure designed for easy field mapping.

Key fields for dashboards:
- `events.totalCount` — Total events in the window
- `strikes.issued` — Strikes issued in the window
- `removals.total` — Downloads removed
- `removals.byReason.AllFilesBlocked` — Malware removals (combine with `AtLeastOneFileBlocked`)
- `cleaned.total` — Downloads cleaned after meeting seeding goals
- `searches.triggered` — Seeker searches triggered
- `jobs.byType.<JobType>.failed` — Failed job runs
- `jobs.byType.<JobType>.nextRunAt` — Next scheduled run
- `health.downloadClients[0].isHealthy` — First download client health
- `health.arrInstances[0].isHealthy` — First arr instance health

</ConfigSection>

<SectionTitle>Deprecated: v1</SectionTitle>

<ConfigSection
  title="GET /api/stats (deprecated)"
>

<Warning>
`/api/stats` is deprecated in favour of `/api/v2/stats` and is scheduled for removal after **2026-08-01**. Responses include `Deprecation: true`, a `Sunset` date, and a `Link` header pointing to the successor version.
</Warning>

The v1 endpoint returns a similar `events` / `strikes` / `jobs` / `health` structure plus optional `recentItems`, controlled by these query parameters:

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `hours` | integer | `24` | Timeframe in hours (range: 1–720) |
| `includeEvents` | integer | `0` | Number of recent events to include (range: 0–100) |
| `includeStrikes` | integer | `0` | Number of recent strikes to include (range: 0–100) |

**Migrating to v2:**

| v1 field | v2 replacement |
|----------|----------------|
| `strikes.totalCount` | `strikes.issued` |
| `strikes.byType` | `strikes.byType` (now derived from strike events) |
| `strikes.itemsRemoved` | `removals.total` |
| *(malware — not in v1)* | `removals.byReason.AllFilesBlocked + removals.byReason.AtLeastOneFileBlocked` |
| `events.*` | `events.*` (unchanged) |
| `jobs.byType` | `jobs.byType` (unchanged) |
| `health` | `health` (unchanged) |

v2 also adds `removals.byReason`, `cleaned`, `searches`, `windowHours`, an `includeDryRun` option, and a bucketed `timeline` endpoint. v2 does not return `recentItems` — use the Events API for recent event/strike records.
