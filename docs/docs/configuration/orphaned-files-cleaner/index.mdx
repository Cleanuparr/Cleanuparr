# Orphaned Files Cleaner

The Orphaned Files Cleaner scans configured directories for files and folders that are no longer tracked by any active torrent in your download client, and moves them to a dedicated orphaned directory. Optionally, it can purge entries from that directory after a configurable number of days.

<ElementNavigator />

<div className={styles.documentationPage}>

<Important>
These settings require a download client to be configured.
</Important>

<div className={styles.section}>

<ConfigSection
  title="Enable Orphaned Files Cleaner"
>

When enabled, the Orphaned Files Cleaner will run according to the configured schedule, scanning your directories for files not claimed by any active torrent.

</ConfigSection>

<ConfigSection
  title="Scheduling Mode"
>

Choose how to configure the Orphaned Files Cleaner schedule:
- **Basic**: Simple interval-based scheduling (every X minutes/hours)
- **Advanced**: Full cron expression control for complex schedules

</ConfigSection>

<ConfigSection
  title="Cron Expression"
>

Enter a valid Quartz.NET cron expression to control when the Orphaned Files Cleaner runs.

**Common Cron Examples:**
- `0 0 * ? * * *` - Every hour
- `0 0 */6 ? * * *` - Every 6 hours
- `0 0 0 ? * * *` - Every day at midnight

</ConfigSection>

</div>

<div className={styles.section}>

<SectionTitle>Directories</SectionTitle>

<ConfigSection
  title="Scan Directories"
>

A list of absolute paths to scan for orphaned files. Each top-level entry inside these directories is checked against your active torrents' save paths. If no active torrent claims an entry, it is considered orphaned.

**Examples:**
```
/data/downloads/completed
/data/downloads/cross-seed
```

<Important>
Each path must be accessible by Cleanuparr. If running in Docker, make sure to mount the directories accordingly.
</Important>

</ConfigSection>

<ConfigSection
  title="Orphaned Directory"
>

The directory where orphaned files and folders are moved. Leave empty to log orphaned entries without moving them (dry-run mode).

**Example:**
```
/data/downloads/orphaned
```

<Note>
The orphaned directory itself is never scanned for orphans.
</Note>

</ConfigSection>

</div>

<div className={styles.section}>

<SectionTitle>Filters & Safety</SectionTitle>

<p className={styles.sectionDescription}>
  These settings help prevent accidental moves by adding age checks, glob-based exclusions, and a per-run cap.
</p>

<ConfigSection
  title="Exclude Patterns"
>

Glob patterns for file or folder names that should never be considered orphaned, even if no active torrent claims them. Matching is case-insensitive.

**Examples:**
```
*.nfo
*.txt
.DS_Store
Thumbs.db
```

<Note>
Patterns are matched against the entry name only (not the full path). Use `*` as a wildcard for any characters.
</Note>

</ConfigSection>

<ConfigSection
  title="Min File Age"
>

Minimum age in minutes a file or folder must have before it can be considered orphaned. This protects files that are actively being downloaded or just finished.

Set to `0` to disable age filtering (default).

**Example:** Set to `60` to ignore files modified less than an hour ago.

</ConfigSection>

<ConfigSection
  title="Max Moves per Run"
>

Maximum number of orphaned entries to move in a single run. Acts as a safety cap to prevent accidental mass moves.

Defaults to `50`. Set higher if you need to process large batches.

</ConfigSection>

<ConfigSection
  title="Purge Orphaned After"
>

Number of days after which entries in the Orphaned Directory are permanently deleted. Leave empty to keep orphaned files indefinitely.

**Example:** Set to `30` to automatically purge orphaned files older than 30 days.

<Warning>
Files deleted by this option are permanently removed from disk. Make sure you have reviewed the contents of your Orphaned Directory before enabling this setting.
</Warning>

</ConfigSection>

</div>

<div className={styles.section}>

<SectionTitle>Path Remapping</SectionTitle>

<p className={styles.sectionDescription}>
  Use these settings when Cleanuparr and your download client see different paths for the same files — common in Docker environments.
</p>

<ConfigSection
  title="Download Directory Source and Download Directory Target"
>

The path prefix as reported by the download client (`Source`) and the corresponding path accessible to Cleanuparr (`Target`). Used to translate torrent save paths so Cleanuparr can correctly determine which files are claimed.

Leave both fields empty if no path translation is needed.

**Docker example:**

```yaml
services:
  qbittorrent:
    volumes:
      - /host/data/downloads:/downloads   # qBittorrent sees /downloads

  cleanuparr:
    volumes:
      - /host/data/downloads:/data/downloads  # Cleanuparr sees /data/downloads
```

In this case:
- **Download Directory Source**: `/downloads`
- **Download Directory Target**: `/data/downloads`

qBittorrent reports save paths starting with `/downloads`. Cleanuparr translates them to `/data/downloads` before comparing against the scan directories.

</ConfigSection>

</div>
